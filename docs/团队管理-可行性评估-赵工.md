## 二、核心难点技术分析

### 2.1 飞书群消息自动同步到看板

**问题：** 童七七的方案期望用户在群里发消息（如"帮赵工加个任务"）就能自动创建任务到看板。

**Feishu 平台限制：**  
- ✅ 用户（自然人）@Bot → 触发事件 → Bot 收到消息  
- ❌ Bot @ Bot → **不触发事件**（Feishu 平台级限制，不可绕过）  
- ❌ 无 @ 的群消息 → 不触发 Bot 事件  

**可行性评估：** ✅ **可以，但需要约定操作方式**

**方案 A（推荐）：指定枢纽 Bot + 自然语言指令**  
```
用户（自然人）→ @蒋小花 "帮赵工加个任务：做7月内容日历排期，7月10日截止"
                      ↓ （Feishu 事件触发）
                  蒋小花 Bot 收到消息
                      ↓ （Hermes Agent 解析 NL→结构化）
                  POST /api/projects/{pid}/tasks
                      ↓ （飞书通知）
                  群内收到: "🎯 新任务创建: 做7月内容日历排期"
```
- **可行性：** 100% 可行。用户 @Bot 触发事件，这是 Feishu 标准功能  
- **NL 解析：** Hermes Agent 本身运行 DeepSeek LLM，具备自然语言→结构化能力  
- **关键：** 需要在蒋小花（default profile）的 Agent 中配置一个自定义工具或 Action，专门处理"帮我加个任务"意图  
- **Token 成本：** 每次解析约 200-500 tokens，几乎可忽略  

**方案 B（降级）：结构化指令格式**  
```
用户 → @蒋小花 "+任务 标题=7月内容日历 负责人=赵工 截止=07-10 优先级=高 项目=小红书"
```
- 不需要 LLM 解析，正则即可处理  
- 学习成本略高  
- **推荐作为 A 的备选**，当 NL 解析失败时 fallback  

**风险：** 用户忘记 @Bot 则消息不会触发 → 需要团队习惯培养

### 2.2 6 个 Agent 的任务状态实时同步

**问题：** 当看板上状态变更，6 个 Agent 如何同步感知当前状态？

**方案对比：**

| 方案 | 实现方式 | 实时性 | 复杂度 | 推荐 |
|------|---------|--------|--------|------|
| **① 群 Webhook 广播（现有）** | 后端 `feishu.py` 已有。任务变更时 `asyncio.create_task(notify_xxx())` 推到群 Webhook | 秒级 | ⭐ 最简 | ✅ **推荐** |
| ② Event Stream (SSE) | 后端加 SSE endpoint，前端 EventSource 连接 | 毫秒级 | ⭐⭐⭐ 中 | ❌ MPA 场景无必要 |
| ③ WebSocket | FastAPI WebSocket，需新增连接管理和心跳 | 毫秒级 | ⭐⭐⭐⭐ | ❌ 过度设计 |
| ④ 前端轮询 | 前端每隔 10s 轮询 `/api/team/board` | 10秒级 | ⭐⭐ 简单 | ❌ MPA 页面刷新即可 |

**推荐方案：群 Webhook 广播**  

架构图：
```
看板操作（Web UI 拖拽/点击）
        ↓
PUT /api/tasks/{id}/reassign  （或 /status）
        ↓
后端更新数据库 → 自动计算负载 → fire-and-forget 飞书通知
        ↓                                            ↓
   返回前端（刷新）                          群 Webhook 接收卡片消息
                                              ↓
                                       6 个 Bot 在群里看到更新
```
- **不需要新服务**，所有基础设施已就位  
- 现有 `feishu.py` 已有 `notify_task_created`, `notify_task_updated`, `notify_task_status_changed`  
- 只需新增 `notify_task_reassigned()`（方案中已设计）  
- **优点：** 无新增运维成本，零额外基础设施  
- **缺点：** 不是实时双向同步，而是"最终一致"——但对看板场景足够  

**关于"实时同步"的澄清：**  
"实时"在这里的含义应是"操作后立即看到"，而非 WebSocket 级的毫秒刷新。当前 MPA 架构下，页面刷新时重新加载数据就是"实时"。若要追求毫秒级更新，需要将 MPA 改造为 SPA（工作量 3-5 倍），**当前阶段不建议**。

### 2.3 看板操作反向同步到飞书

**问题：** 在 Web 看板上操作（拖拽分配、状态切换）需要通知飞书群。

**评估：** ✅ **已经实现，仅需增量修改**

**已存在：**
```

notify_task_updated()      — 任务更新（含变更内容）

```

**需要新增（方案中已设计）：**
```python
async def notify_task_reassigned(task, operator, from_user, to_user):
    """任务重新分配通知"""
    # 卡片内容: 🔄 任务重新分配
    # 「7月内容日历排期」从 童七七 → 蒋小花
```
- 工作量：约 1-1.5 小时  
- 复用现有 `_build_card()` + `_send()` 基础设施  

**触发场景：**
| 场景 | 触发点 | 数据流 |
|------|--------|--------|

### 2.4 消息解析（"帮赵工加个任务"→结构化数据）

**问题：** 自然语言消息 → 结构化的 API 请求数据。

**评估：** ✅ **Hermes Agent 原生能力，零额外成本**

**解析流：**
```
用户消息: "@蒋小花 帮赵工加个任务"
                        ↓
蒋小花 Bot → Hermes Gateway → Agent (DeepSeek LLM)
                                    ↓
    Agent 通过 tool calling 理解意图：
    {
      "intent": "create_task",
      "title": "（自动提取）",
      "assignee": "赵工",
      "priority": "medium",       // 默认
      "project": "（可追问或默认）"
    }
                                    ↓
    Agent 调用 HTTP API → POST /api/projects/{pid}/tasks
```

**LLM 解析粒度：**
| 输入示例 | 解析结果 |
|---------|---------|
| "帮赵工加个任务：做7月内容日历" | title="做7月内容日历", assignee="赵工" |
| "加个高优先级任务：修复登录Bug，交给赵工" | title="修复登录Bug", priority="high", assignee="赵工" |
| "@蒋小花 帮我把'用户反馈整理'改成进行中" | intent="update_status", task="用户反馈整理", status="in_progress" |
| "帮我查一下七七现在有几个任务" | intent="query_tasks", target="童七七" |

**风险与对策：**
| 风险 | 说明 | 对策 |
|------|------|------|
| LLM 幻觉解析 | 提取了错误的关键信息 | 增加确认卡片 "即将创建任务：xxx，确认请点 ✅" |
| 指代不清 | "这个任务" 不知道指哪个 | Agent 追问：@"蒋小花 是哪个任务？" |
| 任务所属项目 | 用户未指定项目 | 可设定默认项目 + 允许智能推测 |
| Token 成本 | 每次 NL 解析约 200-500 tok | 约 ¥0.001/次（DeepSeek 价格），可忽略 |

## 四、不用增加外部服务能否实现？

### 基本结论：✅ **可以**，以下功能零额外服务：

| 功能 | 实现方式 | 外部依赖 |
|------|---------|---------|
| Web 看板页面 | FastAPI + MPA HTML | 无 |
| 后端 API | SQLAlchemy 查询现有表 | 无 |
| 飞书出站通知 | Webhook（已配置） | 无 |

### 需要额外条件的场景：

| 功能 | 条件 | 说明 |
|------|------|------|
| Bot 间自动同步状态 | ❌ **Feishu 平台限制** | Bot @ Bot 不触发事件，无法绕过 |

## 七、最终结论

### 可行 ✅

**团队管理模块在现有架构下完全可行，不需要增加任何外部服务。**

### 核心要点

1. **"Bot 不能 @ Bot" 不是障碍** — 人 @ Bot 才能触发，而用户是自然人，操作流程完全走通
2. **"实时同步" 通过群 Webhook 广播实现** — 秒级通知，对看板场景足够好
3. **"反向同步到飞书" 已有完整基础设施** — 只需新增一个通知函数
4. **"NL 消息解析" 由 Hermes Agent 的 LLM 能力天然解决** — 零额外成本

### 必须接受的限制

| 限制 | 能否解决 | 说明 |
|------|---------|------|
| 不能监听无 @ 的群消息 | ❌ 飞书平台限制 | 用户必须 @Bot |
| 不能 Bot @ Bot 自动触发 | ❌ 飞书平台限制 | 不存在技术绕过 |
| MPA 页面不能毫秒级实时更新 | ❌ 架构限制 | 如需此能力需改为 SPA（不推荐当前做）|

### 一句话总结

> **技术可行，架构代价为零，核心难点可控，工期 4 天，主要风险在拖拽交互和甘特图的手写实现上。**
