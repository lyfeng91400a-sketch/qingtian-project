## 二、需要新增的功能

### 2.1 必要功能（P0）

| 功能 | 说明 |
|------|------|
| **多项目看板切换** | 侧边栏列出所有项目，点击直接切换看板，无需回到项目列表再进 |
| **任务 CRUD** | 新建任务（标题、描述、优先级、负责人、截止日期、项目归属）、编辑内容、删除确认、状态拖拽/点击切换 |
| **项目 CRUD** | 新建项目（名称、描述、颜色标识）、编辑、删除（含确认弹窗） |
| **穗穗念项目数据** | v1.0（已开发）完成的任务列表 + v1.1（规划中）待办任务，完整录入 |
| **小红书博主排期导入** | 第二周内容排期以结构化数据导入，每个排期项转为看板任务 |

| **任务筛选/排序** | 按优先级、负责人、截止日期筛选当前看板 |

## 四、技术路线推荐

### 方案对比

| 维度 | **纯前端 HTML（v2 采用）** | 转小程序 | 转 Web App |
|------|--------------------------|---------|------------|
| **开发周期** | 1-2 天 | 2-3 周 | 1 周 |
| **部署成本** | 零（直接打开文件） | 需注册/审核 | 需服务器/域名 |
| **团队内部使用** | ✅ 最佳 | ❌ 过度 | ⚠️ 可接受 |
| **数据持久化** | localStorage（够用） | 云存储 | 需自建后端 |
| **后续扩展** | 可逐步加后端 API | 生态封闭 | 灵活但成本高 |
| **原型验证成本** | ✅ 最低 | ❌ 最高 | ❌ 偏高 |

### 推荐路线

```
Phase 1（当前）→ 纯前端 HTML v2
  ├─ 静态原型 + localStorage 本地存储
  ├─ 所有功能可交互可体验
  └─ 团队成员可直接打开 .html 使用评估

Phase 2（开发阶段）→ 看场景选型
  ├─ 如果是个人/小团队使用：纯前端 + IndexedDB 足够了
  ├─ 如果是多端需求（移动端访问）：转为 PWA 渐进式 Web App
  └─ 如果需要多人协作/云端同步：加后端 API（Node.js + SQLite）

Phase 3（产品化）→ 根据真实用户反馈决定
  └─ 小程序/原生 App 等需要用户量支撑再投入
```

### 本次 v2 定位

**纯前端 HTML，所有数据存 localStorage，零后端依赖。**
团队成员/用户拿到 .html 文件双击即可使用，是评估期最高效的形态。

## 六、结论

**v1 → v2 是一次"从演示骨架到可用原型"的关键升级。** v1 展示了晴天看板的视觉调性和布局思路，但没有真实数据、没有交互闭环、不能实际使用。v2 从"可以看"变成"可以用"——录入两个真实项目（穗穗念、小红书博主）的数据，补齐任务/项目 CRUD，通过 localStorage 让数据不丢失，再加上批量导入的能力应对后续新项目接入。

> **建议：团队内部用 v2 跑一周实际任务管理，收集真实使用反馈后再决定开发路线。**
