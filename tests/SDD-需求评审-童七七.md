## 三、已发现的 2 个 P0 阻塞性 Bug

### 🐛 P0-Bug-1：导航切换失灵（左侧导航栏切换不了页面）

**现象：** 点击左侧导航栏的"仪表盘"或"项目总览"，但页面视图未切换。

**根因分析（代码审查，推测）：**
- `navTo(view, projectId)` 函数逻辑看起来正确：移除所有 `nav-item` 的 `active` class，然后根据 view 参数切换页面。
- 但在看板视图下，如果在 `nav-projects` 中有高亮的项目按钮，而用户切换到仪表盘后再次点击同一个项目按钮，`navTo('kanban', projectId)` 会执行但可能因为 `currentProjectId` 未变化而导致 **点击事件未按预期触发页面切换**。
- 另外，`showView(viewId)` 依赖 `document.querySelectorAll('.view')` 找到对应 view，但如果某个视图的 HTML 结构有问题（比如 id 拼写错误或不存在），会导致切换失败。

> ⚠️ 以上根因分析未经断点验证，为基于代码阅读的推测，需在开发阶段进一步确认。

**影响范围：** 🔴 阻塞发布。用户无法在仪表盘、项目总览、看板之间自由切换，系统不可用。

**修复建议：**
1. 在 `showView()` 中增加日志或 fallback 处理
2. 确保 `navTo()` 函数在相同 view 下也触发 `render` 刷新
3. 增加 `console.warn` 在 view 找不到时输出错误

### 🐛 P0-Bug-2：待办状态误触"完成"无法恢复

**现象：** 在仪表盘的"我的待办"列表中，点击任务前的勾选框将任务标记为完成（状态变为 done），任务立即从列表消失，用户找不到入口将其恢复回待办状态。

**根因分析（代码审查）：**
- 仪表盘"我的待办"的数据过滤条件为：
  ```javascript
  var myTasks = DATA.tasks.filter(function(t) {
    return t.assignee === 'L' && t.status !== 'done';
  }).slice(0, 6);
  ```
- `status !== 'done'` 导致已完成的任务被过滤掉。
- 虽然 `cycleStatus()` 的逻辑是 `todo → in_progress → done → todo`（循环可恢复），但任务一旦变成 done 就消失在列表中，用户无法再点击它触发 `cycleStatus()`。

**影响范围：** 🔴 阻塞发布。用户误操作后无法就地恢复，体验断裂。

**备选路径（现有工作流可救济）：**
- 用户可跳转到看板页面 → 找到该任务所属项目 → 进入看板"已完成"列 → 点击该任务的状态标签（`cycleStatus()` 触发 `done → todo`）即可恢复。此路径存在但不易发现，且操作链较长（4 步），不满足仪表盘"就地操作"的使用预期。

**修复建议（二选一）：**
1. **推荐方案：** 仪表盘"我的待办"改为显示 `assignee='L'` 的所有任务（含已完成），已完成的显示为勾选状态，允许再次点击恢复。
2. **备选方案：** 在仪表盘增加"查看已完成"的筛选按钮，或将 `cycleStatus` 在 dashboard 上改为"已完成 → 待办"的 toggle 而非循环。
